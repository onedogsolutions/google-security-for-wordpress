## Historical Phase: Phase 14 (Quieter logging + verbose toggle)

### Phase 13 Modifications (v2.2.0)
- Added reCAPTCHA Enterprise **Account Defender** for login/registration. Disabled by default, Enterprise-only. Three pieces: send an anonymous account identifier, interpret the returned labels (log + optional 2FA step-up), and annotate login/2FA outcomes to train Google's site-specific model.
- `GSWP_Verifier`: `verify_token()` gained a 4th `$account_identifier` arg. `build_account_user_info()` attaches `event.userInfo { accountId, createAccountTime }` on login/registration assessments only when Account Defender is on. `resolve_account_id()` keys existing users by their immutable user ID (stable across username/email logins) and registrations by normalised email; `hash_account()` SHA-256s with a lazily-seeded, stored `gswp_account_salt` so only an opaque hash — never email/username/phone — leaves the site. The assessment captures `accountDefenderAssessment` alongside the name; new getters `get_last_assessment_name()` / `get_last_account_labels()`. Identifiers are threaded from every login/registration entry point: WooCommerce (`validate_login` username, `validate_registration` email), WP core (`GSWP_Login` — the resolved `WP_User` on a correct password, else the `log` field; registration email), Xootix (`$creds['user_login']` / email), and PowerPack (`$user_login`).
- New `includes/class-gswp-account-defender.php` (`GSWP_Account_Defender`), constructed with the shared verifier and inert unless `is_active()` (Enterprise + `gswp_account_defender`). Hooks `authenticate` at priority 40 (between the verifier's scoring at 30 and 2FA enforcement at 100) to capture the assessment name + labels into request statics, log any labels, and set a `force_2fa` flag when `gswp_ad_step_up` is on and `SUSPICIOUS_LOGIN_ACTIVITY` is present. Annotates terminal login outcomes via the global `wp_login` (→ `LEGITIMATE` + `CORRECT_PASSWORD`) and `wp_login_failed` (→ `INCORRECT_PASSWORD`, skipping our own `gswp_2fa_required` hold). `annotate()` POSTs to the Enterprise `{name}:annotate` endpoint, fail-open with logging (WC logger, or error_log under WP_DEBUG on non-Woo sites).
- 2FA wiring (`GSWP_Two_Factor`): `enforce_second_factor()` now annotates `CORRECT_PASSWORD` + `INITIATED_TWO_FACTOR` when arming a challenge and persists the assessment name into the pending-login transient (`store_pending_login()` gained an `$assessment_name` param). `ajax_verify()` annotates `LEGITIMATE` + `PASSED_TWO_FACTOR` on success and `FAILED_TWO_FACTOR` on a wrong code, via a small `annotate_2fa()` guard. Held 2FA logins never reach `wp_login` (completion uses `wp_set_auth_cookie`), so there is no double annotation. Step-up forces the challenge for enrolled users only; non-enrolled flagged users are logged, never blocked (manage_options accounts are enrolled by policy).
- New options `gswp_account_defender`, `gswp_ad_step_up` (both default `0`) and lazily-seeded `gswp_account_salt`, wired through `gswp_default_options()`, REST `get_settings`/`update_settings`, and the admin localizer. New React panel `src/components/AccountDefender.jsx` (Enterprise-only; enable toggle + "Require 2FA on suspicious logins" toggle) rendered in `App.jsx`. Assets rebuilt.

## Historical Phase: Phase 12 (reCAPTCHA Enterprise Transaction defense)

### Phase 12 Modifications (v2.1.0)
- Added reCAPTCHA Enterprise **Transaction defense** for the WooCommerce checkout. Turning on the console toggle alone produced nothing because the assessment request carried no transaction data; the plugin now sends `event.transactionData` so Google returns a `fraudPreventionAssessment`. Disabled by default and Enterprise-only (classic v3 has no fraud prevention).
- `GSWP_Verifier`: `validate_checkout()` now builds extra event fields via `build_checkout_event_extra()` and passes them through `verify_token()` → `assess_enterprise_token()` (which now `array_merge`s the extra fields into `event` and sets `fraudPrevention => "ENABLED"`). Transaction data is assembled from the posted checkout fields + `WC()->cart`: billing/shipping `Address` (recipient, lines, locality, administrativeArea, regionCode, postalCode), `paymentMethod`, `currencyCode`, `value`, `shippingValue`, payer `user` (email, plus accountId/creationMs/emailVerified for logged-in customers), and cart `items` (name, per-item value, quantity). Transaction data is only attached when Google's documented minimum is present (billing regionCode + postalCode + paymentMethod); otherwise it is omitted so the request can't 400 and lose the reCAPTCHA score check. The assessment captures `name` and `fraudPreventionAssessment` into instance properties.
- `process_fraud_prevention()` logs the `transactionRisk`, stashes the assessment name + risk in the WooCommerce session (the order does not exist yet at `woocommerce_after_checkout_validation`), and — when the opt-in `gswp_txn_block` setting is on — blocks the checkout when `transactionRisk >= gswp_threshold_txn` (default 0.8). The existing reCAPTCHA score-threshold block is unchanged and always applies.
- New `includes/class-gswp-transaction-defense.php` (`GSWP_Transaction_Defense`) completes the annotation feedback loop. Inert unless WooCommerce is active, the key type is Enterprise, and `gswp_txn_defense` is on. On `woocommerce_checkout_order_processed` it moves the assessment name from the session to `_gswp_assessment_name` order meta (and adds a risk order note); on `woocommerce_order_status_changed` it calls the Enterprise `assessments.annotate` endpoint — completed/processing → `LEGITIMATE` (+ `PAYMENT_GIVEN`), refunded → `FRAUDULENT` (+ `REFUND`), cancelled/failed → `FRAUDULENT`. Annotation is fail-open (logs and ignores errors) and guarded against re-firing via `_gswp_annotated` meta.
- New settings `gswp_txn_defense`, `gswp_txn_block` (both default `0`) and `gswp_threshold_txn` (default `0.8`), wired through `gswp_default_options()`, the REST `get_settings`/`update_settings` route (toggles + 0–1 clamped threshold), and the admin localizer. New React panel `src/components/TransactionDefense.jsx` (rendered in `App.jsx` only when WooCommerce is active; shows an Enterprise-required notice for classic keys) provides the enable toggle, the high-risk block toggle, and the risk-threshold slider. Assets rebuilt with `npm run build`.
- Bumped version to 2.1.0 (main file header, `GSWP_VERSION`, `readme.txt` stable tag + features + changelog, `package.json`, `package-lock.json` root entries).

## Historical Phase: Phase 10 (Fix 2FA popup not opening on cached AJAX-login sites)

### Phase 10 Modifications (v2.0.1)
- Fixed the 2FA code-entry popup failing to open on the live site (`maddogproducts.com`) while working on staging (`maddog.onedog.solutions`). Symptom: an enrolled user's login via the Login/Signup Popup (Xootix) AJAX form was correctly held server-side (our `WP_Error` text appeared inside the Xootix popup), but the authenticator modal never rendered (no `<div>`, no console error), locking the user out.
- Root cause: `assets/js/gswp-2fa-modal.js` only opened the modal on a full-page reload (flag cookie already present) or after catching a native `submit` event on a password form, after which it polled the flag cookie for ~10s. The live site runs **FlyingPress** (page cache + "Delay JavaScript"); with the modal script deferred until user interaction and the Xootix login submitting over AJAX, the `submit`-triggered poll never started, so the held-login flag cookie was never observed. Staging has no such optimiser, so the timing happened to work.
- Fix: replaced the `submit`-event-gated poll with a **life-of-page poll of the flag cookie** (`startPolling()`, every 500ms), plus the existing on-load check and a new `pageshow` (bfcache) check. The modal now opens the moment the server's held-login flag cookie appears — independent of which login form/plugin was used, any page cache, or deferred-script timing. `open()` is now idempotent (`isOpen` guard) so repeated polls don't re-focus or rebuild the modal.
- Bumped version to 2.0.1 (main file header, `GSWP_VERSION`, `readme.txt`, `package.json`) so the asset's `GSWP_VERSION` cache-busting query arg forces FlyingPress/CDN/browser caches to fetch the new script. The popup assets are plain files served from `assets/` (not webpack-built), so no rebuild was required.

## Historical Phase: Phase 9 (2FA code-entry popup; settings consolidated into Google Security page)

### Phase 9 Modifications
- Replaced the redirect-based 2FA challenge (which caused an `ERR_TOO_MANY_REDIRECTS` loop and errored on My Account) with a **popup/modal flow that never redirects**. `GSWP_Two_Factor::enforce_second_factor()` hooks `authenticate` at priority 100: for an enrolled user it arms a single-use pending login (HTTP-only token cookie + transient + readable flag cookie) and returns a `WP_Error`, blocking the sign-in uniformly across wp-login.php, WooCommerce My Account, and the PowerPack AJAX login — no auth cookie is issued without the second factor, and nothing is rendered/redirected so front-end forms can't fatal.
- Added a plain-JS popup (`assets/js/gswp-2fa-modal.js` + `assets/css/gswp-2fa-modal.css`, enqueued on wp-login.php via `login_enqueue_scripts` and on logged-out front-end pages via `wp_enqueue_scripts`). It opens automatically when the readable flag cookie is present (full-page-reload logins) or shortly after a password form is submitted (AJAX logins that never reload), collects the code, and completes the login by posting to the `gswp_2fa_verify` admin-ajax endpoint, then navigates to the returned redirect.
- `ajax_verify()` is authorised by the unguessable HTTP-only pending token (not a WP nonce — a page cache such as FlyingPress would serve a stale nonce on logged-out pages). The pending cookies are set `SameSite=Lax` (the CSRF defence), and failed guesses are capped at 5 before the held login is discarded. Replay is still blocked via the last-used TOTP time step.
- Removed the redirect/interstitial machinery (`maybe_start_challenge`, `handle_challenge`, `maybe_resume_challenge`, `show_challenge`, `challenge_url`) and the `login_form_gswp_2fa` / `wp_login` / `template_redirect` hooks.
- Consolidated the 2FA settings into the Google Security React screen and removed the standalone Settings → Two-Factor Auth submenu (deleted `includes/class-gswp-two-factor-admin.php`). The master switch (`tfa_enabled`) and per-role enforcement (`tfa_enforced_roles`) are now read/written through the existing REST settings route (`class-gswp-rest-api.php`) and rendered inside `src/components/TwoFactorNotice.jsx` (toggle + role checkboxes), with the role list passed via the admin localizer (`roles`).

## Historical Phase: Phase 8 (Fix 2FA login flow across all entry points + admin UI tweaks)

### Phase 8 Modifications
- Fixed a fatal error (HTTP 500) when an enrolled user logged in from a non-`wp-login.php` form (e.g. the WooCommerce "My Account" page). `GSWP_Two_Factor::maybe_start_challenge()` previously rendered the interstitial inline by calling `login_header()`/`login_footer()`, which only exist inside `wp-login.php`; on a front-end login they are undefined and fatal. The challenge is now always rendered on `wp-login.php`: `maybe_start_challenge()` clears the freshly issued auth cookie, stashes the in-progress login behind a single-use, HTTP-only cookie token (`gswp_2fa_pending`) backed by a 5-minute transient, and `wp_safe_redirect()`s to `wp-login.php?action=gswp_2fa`.
- Closed a 2FA bypass on AJAX logins (the PowerPack Login Form module). `maybe_start_challenge()` used to `return` early on `wp_doing_ajax()`, leaving the auth cookie in place so the user was logged in with no second factor. It now clears the auth cookie and sets the pending cookie for AJAX logins too; a new `maybe_resume_challenge()` guard (hooked to both `login_init` and `template_redirect`) detects a logged-out visitor carrying a valid pending cookie and redirects them to the interstitial, so the half-finished AJAX login is forced through 2FA on the next page load.
- Replaced the per-user login-nonce mechanism (`create_login_nonce()`/`verify_login_nonce()`, stored in user meta) with the cookie-token + transient "pending login" helpers (`store_pending_login()`, `get_pending_login()`, `clear_pending_login()`, `challenge_url()`, `cookie_path()`). The token is unguessable and HTTP-only, so it can only be minted by a successful password check, and the TOTP code is still required. `show_challenge()` no longer emits hidden `user_id`/`nonce`/`redirect`/`rememberme` fields (the cookie carries that state).
- The post-2FA redirect now honours both `redirect_to` (wp-login.php) and `redirect` (WooCommerce My Account) request fields.
- Admin settings screen (React): `TwoFactorNotice` now shows a "Two-Factor Authentication Settings" button linking to Settings → Two-Factor Auth (new `twoFactorSettingsUrl` localized from `GSWP_Admin`), placed above the "Set up 2FA from your user profile" link. Both button-styled links carry a `gswp-2fa-btn` class with a scoped CSS rule in `src/styles/index.css` forcing white text (including `:hover`/`:focus`), since WordPress admin's own `a` colour rules otherwise won over Tailwind's `text-white`.
